# Pushing data360-app to our shared Git — step by step

_For the dev partner. Goal: get `data360-app/` into our shared repository cleanly, with no secrets or dependencies committed, so you can clone it, run it, and start building._

You'll receive the code as **`data360-app.zip`** (46 files, ~0.5 MB) from Saurabh. Everything needed is inside it; it's already self-contained (includes the original console as a reference and a sample project for testing). A correct `.gitignore` is included, so dependencies and API keys won't be committed.

---

## 0. Prerequisites
- **Git** installed (`git --version`).
- **Node.js 18+** (22.5+ recommended). Only needed to *run* the app, not to push it.
- Access to the shared repo (the clone/remote URL, and push permission).

## 1. Get the files
Unzip `data360-app.zip`. You'll get a folder `data360-app/` containing `server/`, `client/`, `reference/`, `sample-projects/`, the docs, and `.gitignore`.

> If you were handed the folder directly instead of the zip, and it contains a `.git/` directory, **delete that `.git/` folder first** (`rm -rf data360-app/.git`). It's a stray from packaging and must not overwrite the shared repo's history.

## 2. Clone the shared repo
```bash
git clone <SHARED_REPO_URL> shared-repo
cd shared-repo
```
(If you already have it cloned, `cd` into it and `git pull` first.)

## 3. Place the app in the repo
Pick ONE:

**A — the shared repo is empty / this app is the whole repo:** copy the *contents* of `data360-app/` into the repo root.
```bash
cp -R /path/to/data360-app/. .
```

**B — the shared repo already has other projects (recommended for a shared repo):** drop it in as a subfolder.
```bash
cp -R /path/to/data360-app ./data360-app
```
> If you use option B and the shared repo already has a root `.gitignore`, that's fine — the one inside `data360-app/` still applies to everything beneath it.

## 4. Sanity-check what will be committed (important)
From the repo root:
```bash
git add -A
git status --short
```
Confirm the list looks like source only. Then explicitly verify **nothing sensitive or heavy** is staged — all of these must return nothing:
```bash
git ls-files | grep -E 'node_modules|/dist/|settings\.local\.json|\.sqlite'
```
If that command prints any path, **stop** — the `.gitignore` isn't being applied (usually because you copied a stray `.git` or a pre-existing `node_modules`). Remove the offender and re-run. You should end up with ~46 files (docs, `client/src`, `server/src`, `reference/`, `sample-projects/`, bundled `server/data/grounding.md` + `connectors.json`, and the lockfile).

## 5. Commit and push
```bash
git commit -m "Add Data 360 App — React + Node rebuild (scaffold + reference + build prompt)"
git push origin main        # or your branch name
```
> Team convention: consider pushing to a branch (e.g. `feat/data360-app`) and opening a PR rather than straight to `main`.

## 6. First run after clone (verify it works)
```bash
cd data360-app        # or repo root if you used option A
npm run setup         # installs client deps (creates client/node_modules — git-ignored)
npm run build         # builds the client
npm start             # serves app + API at http://127.0.0.1:4370
```
Open http://127.0.0.1:4370. On a fresh clone the app auto-detects the bundled **`sample-projects/`**, so you'll see **NorthPeak Retail (Demo)** immediately. To use the AI assistant, open **Settings**, choose a provider, and paste your own API key (it's stored locally in `server/data/settings.local.json`, which is git-ignored — never commit it).

## 7. Start building
Read, in order: `HANDOVER.md` → `MIGRATION.md` → `README.md`. Then open **`CLAUDE-BUILD-PROMPT.md`** and paste its prompt into Claude Code (run from the repo root) to replicate the remaining tabs from `reference/console.reference.html`. Work one tab at a time and commit per tab.

---

## Golden rules
- **Never commit API keys.** They live only in `server/data/settings.local.json` (git-ignored). If you ever see it in `git status`, do not commit — check the `.gitignore` is present.
- **Don't commit `node_modules/` or `client/dist/`** — they're regenerated by `npm run setup` / `npm run build`.
- **`reference/` and `sample-projects/` are committed on purpose** — they're the UI/logic spec and test data.
- Keep the backend dependency-free; see `HANDOVER.md §9`.
